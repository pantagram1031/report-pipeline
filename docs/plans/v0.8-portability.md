# v0.8 — Portability wave (any machine, any OS, any coding-capable agent)

Status: IN PROGRESS (started 2026-07-13, follows the v0.7 gate-integrity wave).
Operator directives: (1) the pipeline must run from a plain GitHub clone on any
machine, without the HWP document adapter installed; (2) Linux support; (3) any
provider/agent with coding ability can drive it (not just one vendor's harness);
(4) the document assembly tooling we built ships publicly (it lives in the
separate hwp-master repo — its unpushed fixes go up as part of this wave);
(5) the humanization engine gets more precise.

## W1 — Pluggable document backends (run WITHOUT hwp-master)

Today stage 5 assumes the HWP adapter. Change: a document-adapter contract with
three shipped tiers, chosen in `build.yaml` (`doc_backend:`):

- `bundle` (zero-dependency, ALWAYS available): the deliverable is the frozen
  bundle itself — content.md (validated build grammar), figures/, provenance,
  a rendered single-file HTML preview (stdlib only). Stage 5 becomes "package +
  render preview"; verify_format skips; the 4.5 gate is unchanged. This is the
  any-machine floor.
- `docx` (optional extra `pip install .[docx]`, pure-python python-docx):
  content.md → styled DOCX (headings, EQ as text/OMML-lite fallback, figures
  embedded, tables). PDF conversion left to the user (LibreOffice note).
- `hwp` (existing): unchanged, Windows + Hancom + hwp-master.

Mechanics: `pipeline/scripts/doc_backend.py` (dispatch + shared packaging),
`pipeline/adapters_impl/docx_backend.py`, stage-5 playbook forked by backend,
stages.yaml unchanged (stage 5 has no script gate). Tests: bundle backend fully
tested; docx tests skip when python-docx absent.

## W2 — Linux / any-machine bootstrap

- Audit + fix Windows-isms: sync_local lock (held-fd delete-block is Windows
  semantics → POSIX branch uses fcntl.flock on the same fd; keep O_EXCL create),
  path handling, cp949 assumptions (POSIX consoles are UTF-8; keep the ladder).
- `scripts/bootstrap.py`: clone-and-go — checks Python ≥3.10, creates the
  private profile skeleton (`setup_profile.py` non-interactive path), registers
  neutral default packs, runs a smoke (new_report demo → resume → check on a
  synthetic gate), prints what optional extras enable (docx/studio/hwp).
- CI matrix: ubuntu + windows jobs (same suite; COM-free by design).
- README: quickstart per OS; explicit "no HWP needed for the bundle backend".

## W3 — Agent-neutral operation (any coding-capable model)

- `adapters/generic/ORCHESTRATOR.md`: a single self-contained bootstrap prompt
  any agent can ingest — the resume→playbook→follow→gate loop, the guardrails
  (human gates never self-approved; script gates only via `check`; state only
  via CLI; non-destructive forms), role table with "inline if you have no
  subagents" fallbacks, and the vision caveat (text-only models must not fill
  visual rubrics).
- `adapters/codex/AGENTS.md`: drop-in for Codex-style harnesses (auto-read
  entrypoint), pointing at the same loop.
- Master doc: section links; SKILL files stay for Claude Code (already shipped).

## W4 — Assembly tooling public (hwp-master upstream)

In the separate hwp-master checkout: reword the one unpushed commit whose
message contains personal context; commit the uncommitted fixes (eqn.py brace
auto-scripting T13, build_report/com_backend/layout_qa updates, ops_schema.md);
run its test suite; push to origin/master. Rigorloom side: the hwp adapter doc
points at concrete script entrypoints (fill_report --loop as the single path).

## W5 — Humanization v3 (더 정교화)

- Pack-driven voice: humanize/naturalness prompts get the RESOLVED prose_rules
  pack content injected at runtime (from profile root — never stored in the
  workspace), so rewrites steer toward the operator's actual rules instead of
  generic "natural Korean".
- Deterministic pre-pass: before any LLM round, check_style findings are
  attached to the prepare payload as per-paragraph hints (rule id + span), so
  the rewriter fixes known violations first — fewer rounds, less drift.
- Backend-configurable workers: reviewer/rewriter roles resolve their CLI from
  the backends pack (argv arrays) instead of assuming one harness; a no-LLM
  degraded mode = deterministic checks only + hold_and_report.
- Fidelity gate unchanged (prose_fidelity remains the hard lock); add a
  round-over-round no-progress detector (same violations twice → stop, report).

## Sequencing

W4 first (independent, unblocks "scripts on GitHub"), then W1+W2+W3+W5 in
parallel (disjoint files), then: full suite + privacy scan + independent
adversarial review round + E2E on Linux-like paths (CI), PR to master after the
v0.7 PR merges.

Non-goals (still deferred): release attestation, claim tokens, review graph
(see v0.7-hardening.md); porting the HWP COM layer itself off Windows
(impossible — Hancom is Windows-only; portability = not needing it).
